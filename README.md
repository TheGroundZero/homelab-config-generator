# Podman homelab with Caddy subdomains

## Quick start

This homelab uses a **Python-based batch scaffolder** to generate all infrastructure from a single manifest file:

```sh
# Generate the Quadlets, Caddy site configs, and example env templates from the batch manifest
uv run python3 scripts/scaffold-homelab-batch.py --manifest scripts/apps.yaml --force

# Or do a dry-run first to validate
uv run python3 scripts/scaffold-homelab-batch.py --manifest scripts/apps.yaml --dry-run

# Reload systemd and enable services
systemctl --user daemon-reload
systemctl --user enable --now caddy
systemctl --user enable --now code
systemctl --user enable --now auth
systemctl --user enable --now homeassistant
systemctl --user enable --now mqtt
systemctl --user enable --now zigbee
systemctl --user enable --now esphome
systemctl --user enable --now speedtest
systemctl --user enable --now postgres
```

The batch manifest (`scripts/apps.yaml`) defines all 9 services with their images, ports, volumes, networks, and dependencies. See [Batch manifest configuration](#batch-manifest-configuration) below for customization.

## Podman config

All Quadlet container files are generated by the Python scaffolder into `~/.config/containers/systemd/`.

### Network

- **`homelab.network`** — shared bridge network for infrastructure services (auth, mqtt, zigbee, caddy, code, speedtest, postgres); Home Assistant and ESPHome use `Network=host` for device discovery

### Services

Generated from `scripts/apps.yaml`:

- **`caddy.container`** — Reverse proxy; bridges 8080→80 and 8443→443 for rootless Podman; routes public/local subdomains
- **`auth.container`** — Authentik SSO server; backend for caddy auth gates; depends on postgres
- **`homeassistant.container`** — Home Assistant Core; uses `Network=host` for device discovery; mounts USB devices (`/dev/ttyUSB*`) and Bluetooth adapter (`/dev/hci0`)
- **`mqtt.container`** — Eclipse Mosquitto MQTT broker; shared message bus for IoT devices
- **`zigbee.container`** — Zigbee2MQTT bridge; converts Zigbee to MQTT; depends on mqtt; mounts Zigbee coordinator device (`/dev/serial/by-id/*`)
- **`esphome.container`** — ESPHome device management; uses `Network=host` for device discovery
- **`code.container`** — code-server (VS Code in browser); mounts multiple homelab config directories for editing
- **`speedtest.container`** — Speedtest Tracker; periodic network performance logging
- **`postgres.container`** — PostgreSQL; database backend for Authentik

`AutoUpdate=registry` is set on every container. Enable the timer:

```sh

systemctl --user start podman-auto-update.timer
```

To keep systemd from stopping all of our containers when we log out, we enable _linger_.

## Batch manifest configuration

All services are defined in a single YAML manifest: `scripts/apps.yaml`

### Structure

```yaml
apps:
  - name: caddy
    image: localhost/caddy-custom:latest
    network: homelab.network
    ports:
      - 8080:80
      - 8443:443
    subdomain: caddy
    exposure: public
    auth: no
    devices: []  # Optional: hardware device mappings
    volumes:
      - homelab/caddy/conf:/etc/caddy:z
      - homelab/caddy/snippets:/etc/caddy/snippets:z
      - homelab/caddy/sites:/etc/caddy/sites:z
      - homelab/caddy/data:/data:Z
    requires: []  # Optional: service dependencies
```

### Key fields

| Field | Type | Description |
| --- | --- | --- |
| `name` | string | App name (lowercase); used as container name and service file name |
| `image` | string | Container image URI |
| `network` | string | `homelab.network` (bridge) or `host` |
| `ports` | list | Docker Compose format: `[host:container, ...]` or `[container]` for internal-only |
| `subdomain` | string | Used in Caddy config; typically matches `name` |
| `exposure` | string | `public` or `local` (affects Caddy snippets) |
| `auth` | string | `yes` (Authentik gate), `no`, or `auto` |
| `volumes` | list | Mount format: `homelab/app:/container/path:z` (relative paths with WorkingDirectory=~) |
| `devices` | list | Device mappings (optional): `[host:/container, ...]` for USB/hardware pass-through |
| `requires` | list | Service dependencies: generates `After=` and `Wants=` in systemd unit |
| `userns` | string | `keep-id` (default) or `none` — controls Podman user namespace mapping for the container |

### Customization

Edit `scripts/apps.yaml` to:

- **Add a new app**: Add an entry to the `apps:` list
- **Change port mappings**: Update the `ports:` field (Docker Compose format: `8080:80`)
- **Add device mounting**: Add entries to `devices:` (e.g., Zigbee coordinator, Bluetooth adapter)
- **Adjust network**: Change `network:` to `homelab.network` or `host`
- **Set service dependencies**: Add `requires:` for apps that depend on others
 - **User namespace mapping**: Add `userns:` to control Podman user namespace behavior. Recommended default is `keep-id` (maps the invoking host user's UID into the container user namespace so files the container creates are owned by your host user). Use `userns: none` for services that must be rootful (for example, `Network=host` apps or when accessing devices that require root privileges). If a service run with `keep-id` still has ownership issues (common with images that run as non-root internal UIDs like Postgres), either set `userns: none` or chown the host data directory to the expected container UID before first run (see notes below).

After editing, regenerate all files:

```sh
python3 scripts/scaffold-homelab-batch.py --manifest scripts/apps.yaml --force
```

### Notes on `userns` and data/device permissions

- `userns: keep-id` (recommended default) helps avoid host-owned root files by mapping the calling user's UID into the container. It works well with host users added to device groups (e.g., `dialout` for serial devices) because container processes will be able to access the device using your host group membership.
- `userns: none` disables user namespace remapping and runs the container with full root inside–root on the host mapping. Use this for:
  - `network: host` services that require host networking
  - Containers that must access host devices with kernel-level permissions and cannot rely on host-group membership
- For `postgres` and other DB images: if you use `userns: keep-id` and see permission problems on the data directory, fix by either chowning the data directory to the DB UID or switch that app to `userns: none`:

```sh
# Example: chown postgres data directory to UID 999 inside container
podman unshare chown -R 999:999 homelab/postgres/data
```

### Device mounting

Hardware devices (USB serial, Bluetooth adapters, etc.) can be passed through using the `devices:` field:

```yaml
devices:
  - /dev/ttyUSB0:/dev/ttyUSB0       # Host → Container mapping
  - /dev/hci0:/dev/hci0             # Bluetooth adapter
  - /dev/serial/by-id/usb-*:/dev/zigbee  # Stable symlink for Zigbee coordinator
```

Use stable device symlinks (`/dev/serial/by-id/*`) to avoid device drift on reboot.

Identify devices on your system:

```sh
# Serial/USB devices
ls -la /dev/serial/by-id/

# Bluetooth
hciconfig -a
```

## Caddy config

All of these are under `~/homelab/caddy/`

### `Caddyfile`

Root config — sets the ACME email and imports all snippets and site configs. Update `you@mydomain.tld`.

### Snippets

Under `~/homelab/caddy/snippets/`

- **`local_only.caddy`** — defines `(local_only)`; blocks requests from outside RFC 1918 / VPN ranges with 403
- **`default_headers.caddy`** — defines `(default_headers)`; sets HSTS, `X-Content-Type-Options`, and `X-Frame-Options`
- **`authentik.caddy`** — defines `(authentik)`; forwards auth to Authentik and copies identity headers onto the request

### Caddy env secrets

- Edit the runtime env file at [homelab/caddy/caddy.env](homelab/caddy/caddy.env) for your OVH credentials and any other Caddy-related values.
- The example template lives at [homelab/caddy/caddy.env.example](homelab/caddy/caddy.env.example).
- The generated quadlet loads that runtime file through `EnvironmentFile=homelab/caddy/caddy.env`.
- Keep your real `.env` files local and uncommitted (`*.env` is ignored).

### Sites

All site configurations are generated by the batch scaffolder into `~/homelab/caddy/sites/`:

- **`caddy.caddy`**, **`auth.caddy`**, **`homeassistant.caddy`**, **`zigbee.caddy`**, **`code.caddy`**, **`speedtest.caddy`** — Reverse proxy configs generated from `apps.yaml`
- Site files auto-import snippets and apply auth gates based on manifest `exposure` and `auth` fields

## Enable services

After running the batch scaffolder, reload and enable services:

```sh
systemctl --user daemon-reload

# Enable all generated services
systemctl --user enable --now caddy
systemctl --user enable --now auth
systemctl --user enable --now homeassistant
systemctl --user enable --now mqtt
systemctl --user enable --now zigbee
systemctl --user enable --now esphome

# Verify startup order with dependencies
systemctl --user list-dependencies --all
```

Services with dependencies will start in the correct order (e.g., `postgres` before `auth`, `mqtt` before `zigbee`).


## App onboarding framework

This repository supports two approaches:

1. **Batch scaffolding (recommended)** — Define all services in `scripts/apps.yaml`, then generate all infrastructure at once
2. **Single-app scaffolding** — Use `scripts/scaffold-homelab-app.sh` to add individual apps on demand

### Batch scaffolding (recommended)

Edit the manifest `scripts/apps.yaml`, then generate all files at once:

```sh
python3 scripts/scaffold-homelab-batch.py --manifest scripts/apps.yaml --dry-run  # Validate
python3 scripts/scaffold-homelab-batch.py --manifest scripts/apps.yaml --force   # Generate all files
```

The Python scaffolder generates:
- `.config/containers/systemd/*.container` — Quadlet container definitions
- `homelab/caddy/sites/*.caddy` — Reverse proxy configs, rendered from the Caddy templates in `scripts/templates/caddy/`
- `homelab/*/NAME.env.example` — Environment variable templates

The correct Caddy template is selected automatically based on each app's `exposure` and `auth` fields in the manifest:

| `exposure` | `auth` | Template used |
| --- | --- | --- |
| `local` | `yes` or `auto` | `site.local_authentik.caddy.template` |
| `local` | `no` | `site.local.caddy.template` |
| `public` | `yes` or `auto` | `site.public_authentik.caddy.template` |
| `public` | `no` | `site.public.caddy.template` |

The `{{FQDN}}` and `{{UPSTREAM}}` placeholders are replaced with the app's subdomain+domain and its container-name:port (or `localhost:port` for `network: host` apps like Home Assistant).

See [Batch manifest configuration](#batch-manifest-configuration) for the manifest schema and customization examples.

### Single-app scaffolding (for ad-hoc additions)

Alternatively, use the shell scaffolder to add individual apps without editing the manifest:

```sh
./scripts/scaffold-homelab-app.sh \
  --name myapp \
  --image docker.io/user/myapp:latest \
  --port 8080 \
  --subdomain myapp
```

#### Decision matrix

Classify each app before generating config:

| Exposure | Auth mode | Use when |
| --- | --- | --- |
| local | authentik | Default for internal apps without their own auth |
| local | none | Internal app already behind VPN and/or has strong built-in auth |
| public | authentik | Public app that still requires SSO gate |
| public | none | Public app with its own authentication controls |

#### Naming conventions

- App name: lowercase with dashes (example: `paperless-ngx`)
- Quadlet file: `.config/containers/systemd/<app>.container`
- Container name: `<app>`
- App folder: `homelab/<app>/`
- Caddy site file: `homelab/caddy/sites/<app>.caddy`
- Subdomain label: typically `<app>` unless you intentionally choose something else

#### Caddy templates

The Caddy site templates used by both the batch scaffolder and manual onboarding are in `scripts/templates/caddy/`:

- `site.local.caddy.template` — local-only, no auth (`import local_only` + `import default_headers`)
- `site.local_authentik.caddy.template` — local-only with Authentik gate
- `site.public.caddy.template` — public, no auth (`import default_headers` only)
- `site.public_authentik.caddy.template` — public with Authentik gate

These template files stay in-repo; generated site configs are written to `~/homelab/caddy/sites/`.

Each template uses two placeholders: `{{FQDN}}` (e.g. `ha.mydomain.tld`) and `{{UPSTREAM}}` (e.g. `homeassistant:8123`). To customize a specific site's Caddy config, edit its generated `.caddy` file directly — it won't be overwritten unless you re-run the scaffolder with `--force`.

```sh
./scripts/scaffold-homelab-app.sh \

  --image docker.io/traefik/whoami:latest \
  --subdomain whoami
```

If `--port` is omitted, the scaffold tries to infer it from image `EXPOSE` metadata.

Example with service dependencies:

```sh
./scripts/scaffold-homelab-app.sh \
```

```sh
./scripts/scaffold-homelab-app.sh \

```

```sh
./scripts/scaffold-homelab-app.sh \
  --name code \
  --image ghcr.io/coder/code-server:latest \
  --port 8080 \
  --subdomain code \
  --volume homelab/code/.local:/home/coder/.local:Z \
  --volume homelab/code/.config:/home/coder/.config:Z \
  --volume homelab/homeassistant:/workspaces/homeassistant-config:z
```

#### Supported options

- `--name <app-name>` (required)
- `--image <container-image>` (required)
- `--port <internal-port>` (optional; auto-inferred from image `EXPOSE` metadata when omitted)
- `--subdomain <label>` (required)
- `--domain <base-domain>` (default: `mydomain.tld`)
- `--volume <host-path:container-path[:options]>` (optional; repeatable)
- `--publish <host:container>` (optional; repeatable for multiple mappings)
- `--requires <service1,service2>` (optional; generates `After=` and `Requires=` in `[Unit]`)
- `--selinux-label <Z|z>` (default: `Z`; used as fallback label for generated default volume or any `--volume` mapping without explicit options)
- `--exposure <local|public>` (default: `local`)
- `--auth <auto|yes|no>` (default: `auto`, resolves to `yes`)
- `--network <bridge|host>` (default: `bridge`)
- `--manifest <yaml-file>` (optional; batch mode for multiple app definitions)
- `--continue-on-error` (optional; batch mode only, continue processing next apps after a failure)
- `--dry-run` (optional; validate inputs and guardrails without writing files)
- `--output <text|json>` (default: `text`; `json` currently requires `--dry-run`)
- `--force` to overwrite existing generated files

Guardrails:

- The scaffold rejects duplicate publish ports in the same command.
- The scaffold checks existing quadlets and fails if a requested host port is already used by another service.
- `--publish` cannot be used with `--network host`.
- The scaffold validates dependency names and rejects duplicate or self-referential `--requires` entries.
- The scaffold validates `--selinux-label` and only allows `Z` (private) or `z` (shared).
- Per-volume options in `--volume` are explicit and take precedence over `--selinux-label`.
- When no `--volume` values are provided, the scaffold attempts to infer volume mount points from image metadata and maps them under `%h/homelab/<app>/...`.
- When `--port` is omitted and an image exposes multiple ports, the scaffold auto-selects a preferred web port (`80`, `8080`, `3000`, `8000`, `5000`) for Caddy upstream and auto-generates `PublishPort` mappings for all exposed ports (bridge mode only).
- If image metadata cannot produce a usable port selection, the scaffold fails with guidance to pass `--port` explicitly.

### Batch onboarding manifest

Use `--manifest` to process multiple apps in one run while reusing the same validations as single-app mode.

Manifest file:

- Example file: `scripts/apps.yaml`
- Root key: `apps`
- Supported keys per app entry: `name`, `image`, `port`, `subdomain`, `domain`, `exposure`, `auth`, `network`, `selinux_label`, `publish`, `requires`, `volumes`
- `port` is optional in manifest entries and follows the same image-metadata inference logic as single-app mode.
- `volumes` entries use `host-path:container-path[:options]`; if `:options` is omitted, `selinux_label` is applied as fallback.

Run batch mode:

```sh
./scripts/scaffold-homelab-app.sh --manifest scripts/apps.yaml
```

Validate batch mode without writing files:

```sh
./scripts/scaffold-homelab-app.sh --manifest scripts/apps.yaml --dry-run
```

Validate batch mode and emit machine-readable JSON:

```sh
./scripts/scaffold-homelab-app.sh --manifest scripts/apps.yaml --dry-run --output json
```

Failure behavior:

- Default is fail-fast (batch stops at first app error).
- Add `--continue-on-error` to keep processing remaining apps and return a failed exit code if any app failed.
- Add `--dry-run` to run validations and conflict checks without creating directories or files.
- Add `--output json` with `--dry-run` for CI parsing and app-level result reporting.
- JSON payloads include `schema_version` so CI parsers can pin to a stable output contract.

### Secret handling model

Current implemented baseline:

- Generated quadlets use `EnvironmentFile` and `*.env` files.
- Generated app folders include `app.env.example` placeholders with `CHANGEME` values.
- Real secrets must be stored in local, uncommitted `*.env` files.

Optional hardening path:

- You can migrate sensitive values to Podman secrets and update specific quadlets to consume them.
- Podman secrets are optional in this repository today and are not the default scaffold output.

Practical rule:

- Treat `*.env.example` as documentation templates only.
- Never commit real secret values.

### Onboarding checklist

1. Choose exposure/auth mode using the decision matrix.
2. Create app config using templates or the scaffold script.
3. Fill `homelab/<app>/app.env` from `app.env.example` and set real values.
4. Review volume mappings and environment variables in `.config/containers/systemd/<app>.container`.
5. Reload services: `systemctl --user daemon-reload`.
6. Enable/start app: `systemctl --user enable --now <app>`.
7. Validate Caddy config and reload Caddy.
8. Confirm external access behavior matches expected exposure/auth policy.
9. Revalidate README port map when any Caddy `reverse_proxy` target or quadlet `PublishPort` line changes.

### Validation commands

```sh
# Caddy syntax validation
podman run --rm -v "$HOME/homelab/caddy/Caddyfile:/etc/caddy/Caddyfile:ro,Z" caddy:latest caddy validate --config /etc/caddy/Caddyfile

# systemd user units reload
systemctl --user daemon-reload

# verify service state
systemctl --user status <app>
```

### Verification runbook

Use this short runbook after generating any new app config.

1. Validate Caddy config rendering and syntax.
2. Reload user units and enable/start the new app unit.
3. Confirm the service is healthy and listening as expected.
4. Check routing behavior with the expected policy:
   - local apps should be blocked from non-local sources
   - public apps should be reachable externally
   - Authentik-protected apps should redirect/challenge through Authentik
5. Re-run the scaffold command without `--force` and confirm it refuses overwrites.
6. Run the port inventory check and update README if values drift.

Suggested command sequence:

```sh
# 0) Reconcile port inventory with README table
./scripts/check-port-inventory.sh

# 1) Validate Caddy config
podman run --rm -v "$HOME/homelab/caddy/Caddyfile:/etc/caddy/Caddyfile:ro,Z" caddy:latest caddy validate --config /etc/caddy/Caddyfile

# 2) Reload units and start app
systemctl --user daemon-reload
systemctl --user enable --now <app>

# 3) Check service status and logs
systemctl --user status <app>
journalctl --user -u <app> -n 50 --no-pager

# 4) Check Caddy logs after a test request
journalctl --user -u caddy -n 50 --no-pager
```

### Special cases and scope

- Host network apps (for example Home Assistant): generate with `--network host`, then Caddy upstream should be `localhost:<port>`.
- Hardware access (for example Intel QuickSync): add `AddDevice=` and any required group/device settings in the app quadlet.
- Multi-service dependencies (for example Authentik + Postgres + Redis): create dependent quadlets and network assumptions before enabling the app.

Scope boundaries:

- Included: repo-level config generation, templates, and workflow documentation.
- Excluded: DNS provider automation, certificate account bootstrap, external firewall orchestration, and vault setup.
